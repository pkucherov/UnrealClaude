**Key test patterns to use:**

For session tests: Set `MockHttp->MockResponseBody` to simulate session creation response. After triggering connection sequence (by simulating lifecycle Connected state), verify `MockHttp->PostCallCount` and `MockHttp->LastPostUrl` contain `/session`.

For MCP tests: After Connected transition, verify `MockHttp->LastPostUrl` contains `/mcp` and `MockHttp->LastPostBody` contains `"unrealclaude"` and `"localhost:3000"`.

For execute tests: After session is established, call `ExecuteAsync()` with a config. Verify `MockHttp->PostCallCount` incremented and `MockHttp->LastPostUrl` contains `/session/ses_test123/message`.

For cancel tests: Call `Cancel()`. Verify `MockHttp->LastPostUrl` contains `/abort` (NOT `/cancel`). Verify `MockHttp->bSSEStreamActive` is false.

For permission tests: Simulate SSE data containing a `permission.updated` event via `MockHttp->SimulateSSEData()`. Verify the permission flow triggers (may need to verify that PostCallCount increases when the dialog responds).

**IMPORTANT threading consideration for tests:** FOpenCodeRunner uses `AsyncTask(GameThread)` for dispatching. In test context, game thread dispatches may not execute synchronously. Follow the pattern from Phase 1 decision: "FMockClaudeRunner executes callbacks synchronously to avoid threading in tests." For runner tests, you may need to:
1. Test the synchronous helpers directly (FormatPrompt, CreateConfig, GetDisplayName, etc.)
2. For async flows, verify mock state (PostCallCount, LastPostUrl, LastPostBody) rather than callback execution

**Note on lifecycle integration:** The runner owns `FOpenCodeServerLifecycle` internally. To trigger the Connected state for testing MCP/session/SSE tests, you need the mock to respond to health checks. Set `MockHttp->MockResponseBody` to `{"healthy":true,"version":"0.1.0"}` for health endpoints and `{"id":"ses_test"}` for session endpoints. The lifecycle's `EnsureServerReady()` → `DetectExistingServer()` calls `HttpClient->Get()` for health — configure mocks accordingly.

**PID file consideration:** The lifecycle reads/writes PID files. For tests, either:
- Ensure mock process launcher returns success without real processes
- Clean up any PID files written during tests via `CleanupTestTempDir()`
